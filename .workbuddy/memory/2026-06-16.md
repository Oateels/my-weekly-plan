## 修复：天气/塔罗不停自动刷新 (v36)

- **症状**：用户反映天气和塔罗界面一直不停地自动刷新
- **分析**：详细代码审计未发现明确的递归循环，但可能的触发路径：
  1. `cloudPull` → 合并数据 → `location.reload()` → 重新初始化 → 再次 `cloudPull`
  2. 多个 `setTimeout(fetchWeather, ...)` 排期重叠（500ms + 300ms）
- **防御性修复**：
  1. `fetchWeather`: 添加 `_weatherFetching` 互斥锁 + 10 秒最小间隔（跳过 10 秒内的重复调用）
  2. `initTarot`: 添加 `_tarotInited` 标志，防止重复初始化
  3. `cloudPull` 的 `location.reload()`: 添加 `_syncReloadFired` 会话锁，一个会话最多触发一次 reload

## 修复 v37：彻底解决天气/塔罗自动刷新（sessionStorage 级防重）

- **根因**：v36 使用的 JS 变量 `_syncReloadFired` 在 `location.reload()` 后会重置为 `false`，导致云同步 reload 死循环。且多个 init 块冗余调用 `fetchWeather()`。
- **修复**：
  1. `_syncReloadFired` → `sessionStorage.getItem('__sync_reloaded__')`，跨 reload 持久化
  2. `fetchWeather()` 添加 `sessionStorage('__wx_loaded__')` 级保护，手动 `fetchWeather(true)` 可绕过
  3. `initTarot()` 添加 `sessionStorage('__tarot_inited__')` 级保护
  4. 消除 `reloadAllForWeek` 中冗余的 `fetchWeather()` 调用
  5. `initWeather` IIFE 添加 `sessionStorage('__app_inited__')` 级保护，整体只运行一次
  6. 查询按钮改为 `fetchWeather(true)` 支持手动触发
- SW: v23→v24, 页面 v36→v37, SW 注册 ?v=12→?v=13
- commit `9f307f0` 已推送
