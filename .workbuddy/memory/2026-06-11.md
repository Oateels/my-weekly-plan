## 修复：页面卡死在「检查中」无法加载数据

- **根因1**：`cloudPull`/`cloudPush` 中使用 `fetch` 调用 `api.github.com` 但没有超时（AbortController），GitHub API 连接不上时 fetch 永远挂起，页面 JS 卡死。
- **根因2**：`周计划模板.html` 中已有 `fetchWithTimeout(url, ms)` 但只传 `signal`，没传 headers/method/body，实际无用且被 `cloudPull`/`cloudPush` 忽略。
- **修复**：
  1. 新建正确的 `fetchWithTimeout(url, options, timeoutMs)`（默认 15s 超时）
  2. `cloudPull` 和 `cloudPush`（含 409 重试的嵌套 fetch）全部改用 `fetchWithTimeout`
  3. 页面初始化 sync 代码包装在 try-catch 中，防止同步失败影响整个页面渲染
  4. `service-worker.js` ASSETS 中加入 `周计划模板.html`，确保预缓存
- SW: v19 → v20，版本标记: v27 → v28，commit `be60a83` 已推送。

## 修复：页面仍然卡死（三个根因）

**根因1**（致命）：新增的 `try {` 在 script block 1 开头、`} catch(e)` 被错误地写在了 `</script>` 之后，浏览器解析时 JS 语法错误（Missing catch or finally after try），整个第一个 script block 不执行。同时第三个 script block 残留孤儿 `} catch(e)`（Unexpected token catch）。

**根因2**：`updateSyncStatus` 当用户有 token 但没有同步历史（`wb-sync-last`）时，`st.innerHTML` 从未被更新，"检查中..." 永远残留。

**根因3**：Service Worker 的 `fetch` handler 也没有超时，网络慢时 SW 拦截请求本身可能挂起。

**本轮修复**：
1. 将 `catch` 块移动到 `</script>` 之前（script block 1 内部），删除 script block 3 的孤儿 catch
2. `updateSyncStatus` 增加 else 分支：无同步历史时显示「☁️ 等待首次同步」
3. SW fetch handler 包装在 `new Promise` 中加 15s 超时，超时降级到缓存
4. 添加 5 秒安全网定时器：如果 "检查中" 没被更新，强制显示「☁️ 离线模式」
5. 整个 IIFE 体包在 try/catch 中，JS 错误不再导致页面白屏
6. SW 缓存 v20→v21，页面 v28→v29，SW 注册 ?v=9→?v=10
- commit `de38dad` 已推送。

## 修复：drawSleepChart is not defined (v31)

- **错误信息**：`⚠️ 异常: drawSleepChart is not defined`
- **根因**：代码中初始化 IIFE（line 1126）在第 1132 行**同步**调用 `loadSleepData()` → `drawSleepChart()`，但 `window.drawSleepChart` 的赋值在第 1338 行。执行顺序：调用在前，定义在后，导致 ReferenceError。
- **修复**：`loadSleepData()` 改为 `setTimeout(loadSleepData, 0)`，延迟到同步代码执行完毕后运行（此时 `drawSleepChart` 已定义）。
- 版本 v30→v31，commit `f807f8e` 已推送。

## 修复：入睡时间跨周数据覆盖 (v32)
- **症状**：查看上一周的「入睡时间」后，上一周的数据会覆盖本周的。
- **根因**：`saveSleepData()` 使用 `WEEK_KEY`（始终=真实本周）保存，但 `loadSleepData()` 用 `DISPLAY_WEEK`（当前展示周）加载。`drawSleepChart()` 画图后无条件调用 `saveSleepData()`，导致查看历史周时把历史数据写入本周 key。
- **修复**：`saveSleepData()` 中 `WEEK_KEY` → `DISPLAY_WEEK`，确保读和写使用同一个 key。
- 版本 v31→v32，commit `49feb4a` 已推送。
